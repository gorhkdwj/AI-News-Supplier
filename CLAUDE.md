# CLAUDE.md · ai-news-supplier 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 사용자와 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 이 문서를 먼저 갱신하고 Decisionlog에 이유를 남긴다.

## 0. 프로젝트 정보
- 프로젝트 이름: ai-news-supplier (CLI 명령어: `ains`)
- 목적: AI 소식(커뮤니티·공식 업데이트·핫레포·핫모델·논문)을 API/RSS로 수집해 로컬 SQLite에 축적하고, MCP(stdio)와 CLI를 통해 사용자의 LLM 에이전트에 공급한다. 부가로 학습 가치가 높은 토픽을 룰 기반으로 발굴해 맥락 자료+지시문 패키지를 에이전트에 건네 학습 세션을 설계하게 한다.
- 주요 사용자: LLM 에이전트(Claude Code 등)를 사용하는 개발자. 오픈소스로 공개 예정.
- 최종 산출물: npm 패키지 (CLI `ains` + MCP 서버 `ains-mcp`)
- 마감·제약: 마감 없음(실사용 도구). Node ≥ 22.12(D-006, 0.1.1부터), TypeScript/ESM, Windows 우선 지원. **도구 자체는 LLM을 호출하지 않는다.** 로컬 우선(서버 비용 0원), 코어는 통로(MCP/CLI)에 의존하지 않게 유지.
- 현재 사용자 수준: 초급~중급 (TypeScript/MCP 개념이 익숙하지 않음)
- 원하는 진행 방식: 검증 중시 + 속도 우선 + 실무형 문서화. 새 기술 용어는 처음 나올 때 한 줄로 쉽게 설명한다.

## 1. 대화와 보고 방식
- 모든 응답은 격식 있는 한국어 대화체(합쇼체)로 한다.
- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다.
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.

## 2. 작업 시작 전 확인
- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·변경사항을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.
- 구현 기준 문서: `docs/plans/2026-07-09-ai-news-supplier-plan.md`(확정 계획), `docs/requirements-contract.md`(기준 계약). 두 문서와 어긋나는 구현을 하기 전에 반드시 문서를 먼저 갱신한다.

## 3. 파일 구조 원칙
- `src/` 실행 코드 · `tests/` 테스트와 fixture · `tools/` 개발 보조 스크립트 · `docs/` 문서 · `out/` 실행·임시 산출물(Git 제외).
- `src/` 내부 경계: `src/core`(수집·저장·랭킹·학습 로직)는 `src/cli`/`src/mcp`를 절대 import하지 않는다. 통로는 코어를 호출만 한다.
- `data/` 폴더는 만들지 않는다. 사용자 데이터(SQLite DB, 설정)는 저장소가 아니라 사용자 홈 `~/.ai-news-supplier/`에 둔다.
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙
- 처음부터 전체를 달리지 않고 가장 작은 성공 단위부터 만든다. 구현 순서는 `docs/plans/implementation-plan.md`의 단계(S0~S5)를 따른다.
- 각 단계는 다음을 명확히 한다: 왜 필요한가 / 이전 단계와 연결 / 다음 단계로 전달 / 완료 조건 / 검증 방법 / 실패 시 중단점.
- 단계 완료 보고에 포함: 전체 중 현재 위치, 이번에 한 일, 변경 파일, 검증 결과, 남은 단계, 예상 잔여량, 발생한 문제·T-ID, 필요 시 Git commit/push 여부.
- 프로젝트 고유 불변 규칙:
  - 한 수집 소스의 실패가 전체 수집을 절대 깨지 않는다.
  - MCP 서버 프로세스에서 stdout에는 MCP 전송 외 어떤 출력도 쓰지 않는다(로그는 stderr).
  - 도구 자체에서 LLM API를 호출하지 않는다.

## 5. 기준 계약 문서 원칙
- 구현 전 `docs/requirements-contract.md`를 기준으로 한다: 데이터 스키마 / MCP 도구 입출력 / 지표 수식(hotness, learnScore) / 오류 판정 / 완료 기준.
- 같은 지표가 파일마다 다르게 정의되지 않게 한다. 기준이 바뀌면 코드보다 계약 문서를 먼저 갱신하고 이유를 기록한다.

## 6. 테스트와 검증 원칙
- 영향 범위에 맞춰 검증한다: 단위·통합·입력·오류 상황·경계값 검증, 문서-코드 정합성, README-기능 일치.
- 수집기 테스트는 실제 네트워크가 아니라 녹화된 fixture(저장된 API 응답)를 기본으로 한다. 라이브 네트워크 검증은 수동으로만 수행하고 결과를 보고한다.
- 검증하지 못한 부분은 숨기지 말고 `미검증 범위`로 보고한다. 실패한 테스트를 삭제·완화해 통과시키지 않고 원인을 수정한다.

## 7. 보안·개인정보·비밀정보 원칙
- 다음을 코드·문서·로그·테스트 데이터에 넣지 않는다: API 키, 비밀번호, 액세스 토큰(GitHub/Reddit 포함), 세션 쿠키, 실제 개인정보.
- 사용자 토큰은 `~/.ai-news-supplier/config.json` 또는 환경변수로만 받고, 저장소·로그·오류 메시지에 노출하지 않는다.
- 외부 데이터 수집은 공개 API·RSS·허용된 데이터만 사용하고, 각 API의 rate limit과 이용 약관을 지킨다.
- 비밀정보가 발견되면 임의로 로그를 편집하지 말고, 노출 범위와 안전한 교체·폐기 방법을 사용자에게 보고한다.

## 8. 문서화 원칙
- README에는 실제 구현된 기능만 쓴다. 구현되지 않은 기능을 제공한다고 표현하지 않는다.
- 기획서·기준 계약 문서·코드·테스트·README·Worklog·Decisionlog가 서로 모순되지 않게 관리한다.

## 9. 외부 자료 사용 원칙
- 외부 자료(API 문서, 엔드포인트 사양 등) 사용 시 기록한다: URL / 제목 / 확인일 / 가져온 판단 근거 / 적용 범위.
- 외부 API의 응답 형태·파라미터는 구현 시점에 실물로 재확인한다(계획서의 "구현 시 재확인 항목" 참조).

## 10. Git 원칙
- 주요 단계가 끝날 때마다: 변경 파일 확인 → 불필요한 파일 제외 확인 → 테스트/검증 실행 → commit → push.
- 원격: https://github.com/gorhkdwj/AI-News-Supplier.git (main 브랜치)
- 로그, 비밀정보, 로컬 실행 결과, 임시 산출물(`out/`), `node_modules/`, `dist/`는 Git에 올리지 않는다.

## 11. 로그 기록 규칙

### Worklog.md — 주요 요청마다
모든 주요 사용자 요청이 끝날 때마다 기록한다. 설명 작업이라도 방향·구조·판단에 영향을 주면 기록한다.
```
### W-001 · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### Decisionlog.md — 방향을 바꾸는 결정만
대상: 문제 선정, 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경. 사소한 수정은 Worklog만.
```
### D-001 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### Troubleshootinglog.md — 실제 문제 발생 시
실제 오류·실패·환경 문제·검증 실패·설계 충돌이 발생하면 기록한다. 같은 문제가 반복되면 새 T-ID를 만들기 전에 기존 T-ID를 먼저 확인한다.
```
### T-001 · 문제 제목
**발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치**(+최종 해결) / **재발 방지**
```
