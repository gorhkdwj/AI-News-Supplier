01 · WHY AINS

개발 배경과 특징

흩어진 AI 소식을 에이전트가 다시 사용할 수 있는 로컬 데이터로 만들고, 출처마다 다른 ‘화제성’을 같은 잣대로 섞지 않는 것이 출발점입니다.

PROBLEM 01

소식이 흩어져 있습니다

공식 블로그, GitHub, 커뮤니티와 논문 사이트를 각각 찾아야 하며, 다음 날 같은 조사를 반복하게 됩니다.

PROBLEM 02

점수의 의미가 다릅니다

GitHub 별, HN points, Reddit upvotes와 공식 발표의 중요도는 서로 직접 비교할 수 없는 신호입니다.

SOLUTION

로컬에 모아 유형별로 봅니다

출처 관측을 분리해 보존하고 Community, Official, Repos, Research에 각기 다른 랭킹을 적용합니다.

공개 소스API · RSS · Atom
수집기실패 격리 · 정규화
로컬 DBStory · Sighting · FTS
CLI / MCP같은 코어 기능
사용자 / 에이전트브리핑 · 조사 · 학습
0원별도 서버 운영비
9개MCP 공개 도구 수
14일일반 지표 관측 보존
48시간Reddit 데이터 최대 보존
도구와 지능의 역할을 분리합니다.

ains는 수집·저장·검색·랭킹과 학습 근거 패키징을 담당합니다. 번역, 요약, 설명과 학습 진행은 데이터를 받은 Codex 같은 에이전트가 수행합니다.

02 · GET STARTED

설치와 빠른 시작

npm 레지스트리에서 전역 설치하면 ains와 ains-mcp 명령이 바로 사용 가능해집니다.

필수 환경

Node.js 20 이상과 npm이 필요합니다. Windows를 우선 지원하며 macOS와 Linux에서도 Node.js 환경으로 실행할 수 있습니다.

1

npm에서 전역 설치합니다

shell 
npm install -g ai-news-supplier
2

설치를 확인합니다

shell 
ains --version

소스에서 직접 빌드하려면(개발·기여용) 저장소를 복제한 뒤 npm install && npm run build && npm link를 사용합니다.

3

환경과 DB를 점검합니다

shell 
ains doctor

Node.js 버전, 데이터 경로, DB 무결성, 스키마 버전과 소스별 상태가 표시됩니다. 오류가 있으면 수집 전에 먼저 해결하십시오.

4

첫 데이터를 수집하고 조회합니다

shell 
ains fetch
ains trends --limit 12
ains trends --ranking v2 --channel overview --sort briefing --limit 12
PowerShell에서 여러 소스를 지정할 때

쉼표 목록은 문자열로 묶는 편이 안전합니다. 예: ains fetch --source "hackernews,github"

03 · MENTAL MODEL

핵심 개념

결과와 점수를 올바르게 해석하려면 Story와 출처 관측의 차이를 먼저 이해해야 합니다.

Story

하나의 원문 또는 사건을 나타내는 이야기 단위입니다. ID, 검색 인덱스와 학습 이력의 기준이며 같은 URL의 여러 출처 관측을 하나로 묶습니다.

Sighting

Story를 특정 소스에서 발견한 기록입니다. RSS 항목, HN 글, Reddit 글, GitHub 저장소처럼 출처별 URL·점수·댓글·작성자 정보를 독립적으로 보존합니다.

Metric Snapshot

Sighting의 점수와 댓글 수를 1시간 UTC 버킷으로 관측한 값입니다. 값이 같아도 기록하며 6시간·24시간·7일 성장률의 근거가 됩니다.

채널 포함 내용 주요 판단 신호 기본 보기
overview 네 채널을 균등 할당 후 부족분 재배분 채널별 결과 조합, Story 중복 제거 briefing
community HN, Reddit, DEV.to 추천·댓글 수준, 6h·24h 속도, 시간 감쇠 hot
official 공식 RSS·Atom·정식 Releases 발행 시각 또는 결정적 영향 등급 latest
repos AI 관련 GitHub 저장소 별 증가, 누적 별, 최근 push trending
research 모델, 논문, DEV 외 article 기존 research_hot_v1 hot
채널 간 점수를 비교하지 마십시오.

Repo의 0.8과 Community의 0.8은 같은 의미가 아닙니다. ranking.kind가 같은 결과 안에서 순서와 근거를 해석하십시오.

04 · CLI REFERENCE

CLI 전체 기능

ains <command> --help로 언제든 현재 설치 버전의 옵션을 확인할 수 있습니다.

명령 역할 네트워크·변경 대표 예시
trends 트렌드 조회 TTL이 지난 소스 자동 갱신 ains trends --limit 20
fetch 최신 항목 수집 네트워크·DB 변경 ains fetch --force
search FTS 전문 검색 로컬 조회 ains search "agents"
show Story 상세 조회 로컬 조회 ains show <id>
doctor 환경·DB 진단 로컬 조회 ains doctor
mcp STDIO MCP 서버 실행 에이전트 연결용 ains mcp
learn 학습 후보·세션·기록 일부 자동 갱신·기록 ains learn candidates
history 학습 이력 조회 로컬 조회 ains history
schedule 정기 수집 등록·해제 OS 스케줄 변경 ains schedule status
config 설정 경로·초기화·편집 일부 파일 변경 ains config show
ains trends · 유형별 트렌드 조회

랭킹 버전, 채널과 정렬을 지정해 최근 AI 소식을 조회합니다. 기본적으로 TTL이 지난 소스를 수집한 뒤 결과를 반환하며 --no-refresh로 로컬 DB만 사용할 수 있습니다.

shell 
ains trends --limit 20
ains trends --ranking v2 --channel community --sort hot --hours 48
ains trends --ranking v2 --channel official --sort latest --json
ains trends --ranking v2 --channel repos --sort discovery --no-refresh

옵션: --ranking, --channel, --sort, --limit, --source, --type, --hours, --no-refresh, --json

ains fetch · 최신 데이터 수집

활성 소스를 수집합니다. TTL이 남아 있으면 건너뛰며 --force는 TTL을 무시합니다. 하나의 소스가 실패해도 다른 소스 결과는 저장됩니다.

shell 
ains fetch
ains fetch --source "hackernews,github" --force
ains fetch --quiet
ains fetch --json

--quiet는 스케줄러에서 요약 출력 없이 종료 코드만 확인할 때 사용합니다.

ains search와 ains show · 검색과 상세 조회

search는 SQLite FTS 전문 검색으로 제목·요약 등의 텍스트를 찾습니다. show는 결과 ID로 Story 상세를 조회합니다.

shell 
ains search "mixture of experts" --days 30 --limit 10
ains search "agents" --type "paper,article" --json
ains show <story-id>
ains show <story-id> --json

show --json에는 raw 원본과 Sighting·metric history가 포함될 수 있으므로 공유 전에 내용을 확인하십시오.

ains learn와 ains history · 학습 워크플로

최근 데이터에서 학습 후보를 찾고, 선택한 토픽의 맥락과 에이전트용 학습 지시문을 생성하며, 완료한 학습을 기록합니다. ains 자체가 강의를 생성하거나 LLM을 호출하지는 않습니다.

shell 
ains learn candidates --limit 5 --days 7
ains learn candidates --include-learned --no-refresh --json
ains learn session "RAG" --level beginner --time 30
ains learn record "RAG" --level beginner --time 30 --notes "기초 완료"
ains history --limit 20 --json
ains config, doctor, schedule · 운영 명령
shell 
ains config path
ains config show
ains config init
ains config edit

ains doctor

ains schedule install --every 60
ains schedule status
ains schedule uninstall

config init/edit는 사용자 설정 파일을, schedule install/uninstall은 OS 스케줄을 변경합니다. 자동화에서 실행하기 전에 변경 범위를 확인하십시오.

05 · RANKING V2

트렌드 랭킹

유형마다 “지금 볼 가치가 있는가”를 나타내는 신호가 다르므로 채널별 순위기를 사용합니다.

channel 허용 sort 특징 점수
overview briefing 네 섹션 균등 배치 각 항목의 채널 점수
community hot, latest 참여 수준과 증가 속도 latest는 없음
official latest, important 최신순 또는 영향도 latest는 없음
repos trending, discovery 성장 본선과 신규 발견 분리 discovery는 없음
research hot, latest research_hot_v1 유지 latest는 없음

Repository Trending

  • AI 관련, 비 fork·비 archived
  • 별 100개 이상, 최근 14일 내 push
  • 24시간·7일 기준점 모두 필요
  • 별 증가, 누적 규모와 push 신선도를 결합

생성 14일 이내 저장소는 별 수와 무관하게 discovery에 표시할 수 있지만 trending 본선 점수는 받지 않습니다.

Community Hot

  • 소스 내부 백분위와 절대 로그 수준
  • 추천·댓글의 6시간·24시간 증가 신호
  • 48시간 반감기의 시간 감쇠
  • 상위 결과의 출처 다양성 목표

현재 추천 점수가 null이면 HOT에서는 제외하지만 최신순에는 포함할 수 있습니다. 누락 신호는 나머지 가중치를 재배분합니다.

Official Updates

  • latest: 발행 시각 내림차순, 점수 없음
  • important: 영향도와 커뮤니티 echo
  • critical: 보안, 폐기, 지원 종료, breaking
  • high: 모델·API·SDK 출시, 가격·한도 변경

고객 사례·행사·웨비나는 제품명이 있어도 먼저 낮은 영향 규칙을 적용합니다.

Research

논문, 모델과 DEV.to 이외의 article은 별도 벤치마크 전까지 기존 research_hot_v1 공식을 유지합니다. DEV.to article은 저장 타입을 바꾸지 않고 Community 채널에서 다룹니다.

latest는 점수 없이 발행 시각 중심으로 볼 때 사용합니다.

0.1.0은 v2 shadow 단계입니다.

옵션 없는 ains trends는 legacy를 사용합니다. v2 Repo Trending은 실제 관측으로 24시간·7일 기준점이 쌓이기 전까지 결과가 적거나 없을 수 있습니다. 과거 별 증가량을 임의로 만들어내지 않습니다.

올바른 조합 
ains trends --ranking v2 --channel overview --sort briefing
ains trends --ranking v2 --channel community --sort hot
ains trends --ranking v2 --channel official --sort important
ains trends --ranking v2 --channel repos --sort trending
ains trends --ranking v2 --channel research --sort latest
오류 예시 
ains trends --ranking v2 --channel repos --sort hot
# 오류: Sort hot is incompatible with channel repos. (종료 코드 1)

06 · AGENT CONNECTION

MCP 연결

MCP는 에이전트가 외부 도구를 정해진 스키마로 호출하는 표준입니다. ains는 STDIO 방식을 사용합니다.

HTTPS 주소가 없어도 정상입니다.

ains-mcp는 로컬 자식 프로세스로 실행되고 표준 입력과 출력으로 MCP 메시지를 교환합니다. 원격 Streamable HTTP 서버 URL은 필요하지 않습니다.

연결 전 확인

shell 
ains --version
ains doctor

# Windows PowerShell
Get-Command ains-mcp

# macOS / Linux
command -v ains-mcp

터미널에서 ains-mcp만 직접 실행하면 메시지를 기다리며 멈춘 것처럼 보일 수 있습니다. 이는 대화형 CLI가 아니라 MCP 클라이언트가 연결해야 하는 서버이기 때문입니다.

저장소의 .codex/config.toml에 등록하면 신뢰된 프로젝트에서만 사용합니다.

toml 
[mcp_servers.ains]
command = "ains-mcp"

모든 Codex 프로젝트에서 보이게 하려면 사용자 전역에 등록합니다.

shell 
codex mcp add ains -- ains-mcp
codex mcp list

Claude Code CLI에서 로컬 STDIO 서버를 등록합니다.

shell 
claude mcp add ains -- ains-mcp

PATH 연결이 어려우면 빌드 산출물을 Node.js로 직접 실행할 수 있습니다.

toml 
[mcp_servers.ains]
command = "node"
args = ["C:/absolute/path/AI-News-Supplier/dist/mcp/server.js"]

등록 후 확인 순서

  1. Codex 또는 Claude Code를 완전히 종료합니다.
  2. 다시 실행하고 새 작업을 엽니다.
  3. MCP 목록에서 ains 또는 도구 9개를 확인합니다.
  4. “ains 수집 소스 상태를 확인해줘”라고 요청해 get_source_status 호출을 확인합니다.
목록 표시와 실제 호출 가능 상태가 다를 수 있습니다.

일부 Codex 앱 버전에서는 프로젝트 범위 MCP가 /mcp 화면에 누락되어도 실제 도구가 호출되는 사례가 확인되었습니다. 표시만으로 판정하지 말고 읽기 전용 get_source_status 호출 결과도 확인하십시오.

07 · MCP REFERENCE

MCP 도구 9개

CLI와 같은 코어를 사용하지만 JSON 스키마로 입력과 구조화된 결과를 주고받습니다.

도구 역할 주요 입력 동작
get_trends 트렌드 조회 limit, sources, types, since_hours, ranking_version, channel, sort 자동 갱신
search_news FTS 전문 검색 query, limit, since_days, types 자동 갱신
get_item Story 상세 id 읽기
refresh_sources 수동 수집 sources, force DB 변경네트워크
get_source_status 소스 상태 없음 읽기
get_learning_candidates 학습 후보 limit, since_days, include_learned 자동 갱신
design_learning_session 학습 지시문·근거 topic, level, time_budget_minutes 읽기
record_learning 학습 이력 기록 topic, level, time_spent_min, notes, item_ids DB 변경
get_learning_history 학습 이력 조회 limit 읽기
get_trends · 최신 AI 트렌드

최신 트렌드를 필터링해 반환합니다. 최대 limit=100이며, v2에서는 sections[]와 같은 순서로 평탄화한 items[]를 함께 제공합니다.

arguments 
{
  "ranking_version": "v2",
  "channel": "repos",
  "sort": "trending",
  "since_hours": 168,
  "limit": 10
}
search_news · 전문 검색

최근 데이터를 갱신한 후 축적된 Story를 FTS로 검색합니다.

arguments 
{
  "query": "mixture of experts",
  "since_days": 30,
  "types": ["paper", "article"],
  "limit": 20
}
get_item · Story, Sighting과 지표 이력

Story ID로 원본 raw, 기존 score history, 전체 sightings[]와 Sighting별 metric_history를 조회합니다. 항목이 없으면 found: false입니다.

arguments 
{ "id": "<story-id>" }
refresh_sources와 get_source_status · 수집 운영

refresh_sources는 특정 또는 전체 소스를 수집하고, get_source_status는 활성 여부, TTL, 마지막 성공, 연속 실패와 항목 수를 반환합니다.

arguments 
// refresh_sources
{ "sources": ["hackernews", "github"], "force": true }

// get_source_status
{}
get_learning_candidates · 학습 후보 발굴

여러 소스 등장, 최근 상승과 신규성으로 학습 후보를 찾고 공식·논문·레포·토론 근거를 버킷별로 제공합니다. 기본은 최근 7일, 5개이며 이미 학습한 토픽은 제외합니다.

arguments 
{
  "limit": 5,
  "since_days": 7,
  "include_learned": false
}
design_learning_session · 학습 세션 설계 자료

arguments

08 · RECIPES

작업 예시 10가지

에이전트가 MCP를 인식하면 자연어 요청을 적절한 도구 호출로 연결할 수 있습니다.

자동 호출은 에이전트 환경에 따라 달라집니다.

MCP가 활성화되어 있고 도구 사용이 허용되어야 합니다. 상태 변경이나 네트워크 접근은 에이전트의 승인 정책에 따라 사용자 확인을 요구할 수 있습니다.

4 “막 공개된 신생 AI 저장소를 찾아줘.” get_trends repos/discovery ains trends --ranking v2 --channel repos --sort discovery

  2. 공식·레포·커뮤니티·연구 섹션을 같은 개수로 훑습니다.
  3. ranking.explanation과 출처 URL로 선정 이유를 설명합니다.
  4. 더 볼 항목이 있으면 사용자 동의 후 get_item으로 상세를 조회합니다.

  2. 공식·논문·레포·토론으로 자료를 분류합니다.

  4. 출처 링크와 확인된 사실, 에이전트의 해석을 구분해 보고합니다.
  1. get_learning_candidates로 상위 후보와 추천 이유를 제시합니다.

09 · READ THE RESULT

결과 해석

점수만 보지 말고 coverage, signals와 explanation을 함께 확인해야 순위의 근거가 보입니다.

축약된 결과 예시 
{
  "id": "<story-id>",
  "source": "github",
  "type": "hot_repo",
  "title": "Example AI repository",
  "score_kind": "stars",
  "comments_count": null,
  "discussion_url": null,
  "activity_at": "2026-07-10T01:00:00.000Z",
  "ranking": {
    "version": "v2",
    "channel": "repos",
    "sort": "trending",
    "kind": "repo_trending_v2",
    "position": 1,
    "score": 0.73,
    "coverage": "full",
    "signals": { "star_delta_24h": 420, "star_delta_7d": 1800 },
    "explanation": "24시간·7일 성장과 누적 규모, 최근 push를 반영"
  },
  "hotness": 0.73
}

위 값과 제목은 구조 설명을 위한 예시이며 실제 수집 결과가 아닙니다.

필드 의미 해석할 때 주의할 점
ranking.version legacy 또는 v2 서로 다른 버전의 점수를 직접 비교하지 않습니다.
channel / sort 유형과 보기 방식 채널별로 점수의 의미가 다릅니다.
kind 실제 랭킹 공식 식별자 같은 kind 안에서 비교하는 것이 안전합니다.
position 현재 결과 안의 순서 필터·시간·limit이 달라지면 바뀝니다.
score 해당 랭커의 점수 latest·discovery는 null일 수 있습니다.
coverage 필요 기준점 확보 정도 full이 아니면 성장률 신뢰도가 낮을 수 있습니다.
signals 별·추천·댓글·시간 등 계산 근거 원값과 정규화 신호를 함께 확인합니다.
explanation 사람이 읽는 선정 이유 에이전트 요약에서 근거 문장으로 활용합니다.
hotness 이전 호환 별칭 폐기 예정이며 1.0까지 유지됩니다.
score_kind 원천 점수 단위 points, upvotes, reactions, stars, likes 등입니다.

기본값으로 대부분의 기능을 사용할 수 있고 GitHub·Reddit 한도를 넓힐 때만 인증을 추가합니다.

설정 파일

~/.ai-news-supplier/config.json

데이터 경로

잘못된 설정은 경고 후 기본값으로 폴백합니다.

인증 환경변수

환경변수 용도 필수 여부
GITHUB_TOKEN GitHub Search·Repository·Gemini Releases API 한도 선택
AINS_REDDIT_CLIENT_ID Reddit OAuth client ID Reddit 사용 시 세 값 모두 필수
AINS_REDDIT_CLIENT_SECRET Reddit OAuth client secret
AINS_REDDIT_USERNAME 식별 User-Agent의 Reddit username
AINS_HOME 사용자 데이터·설정 디렉터리 변경 선택
비밀정보를 저장소에 커밋하지 마십시오.

환경변수나 사용자 홈 설정만 사용하십시오. 토큰, client secret, 세션 쿠키, 실제 사용자 데이터와 DB 파일을 이슈·로그·fixture에 붙이지 마십시오.

사용자 정의 RSS

sources.rss.feeds를 설정하면 기본 목록에 추가되는 것이 아니라 기본 피드 전체를 대체합니다. 기본 피드와 사용자 피드를 함께 쓰려면 원하는 기본 항목도 배열에 모두 포함해야 합니다. 각 피드에는 고유 id, 표시용 title, 공개 url이 필요합니다.

12 · LOCAL DATA

데이터와 보안

수집 데이터와 학습 이력은 사용자 컴퓨터에 저장되며 보존 정책에 따라 자동 정리됩니다.

데이터 위치·기간 정책
SQLite DB ~/.ai-news-supplier/data.db 로컬 저장
일반 Story 기본 90일 retentionDays, null이면 영구
일반 metric snapshot 14일 1시간 UTC 버킷
Reddit Sighting·본문·작성자·snapshot 최대 48시간 삭제·removed는 재검증 후 즉시 제거
학습 이력 사용자 DB 후보 중복 방지와 history 조회
v1→v2 백업 data.pre-v2.<UTC>.bak 마이그레이션 전 VACUUM INTO

DB 마이그레이션과 복구

파일 DB를 v2로 처음 열면 같은 디렉터리에 사전 백업을 만들고 무결성·rowid·FTS·학습 이력과 기존 점수 이력을 검증합니다. 백업이나 검증이 실패하면 마이그레이션을 시작하지 않습니다.

  1. 실행 중인 ains, MCP 서버와 스케줄 작업을 모두 종료합니다.
  2. 현재 data.db를 삭제하지 말고 별도 이름으로 보존합니다.
  3. 선택한 data.pre-v2.<UTC>.bak 복사본을 data.db로 둡니다.
  4. 호환 버전으로 실행한 뒤 ains doctor로 무결성을 확인합니다.

13 · READ BEFORE USE

주의사항과 한계

자동 브리핑과 학습에 사용하기 전에 현재 버전의 데이터·랭킹·운영 경계를 확인하십시오.

v2 기준점은 시간이 필요합니다

Repo Trending은 실제 24시간·7일 snapshot을 요구합니다. 설치 직후에는 Discovery와 다른 채널을 먼저 활용하십시오.

최신이 곧 중요함은 아닙니다

Official latest는 점수 없는 시간순입니다. 영향이 큰 변경은 important로 별도 조회하십시오.

외부 API 상태에 영향을 받습니다

rate limit, 응답 형식 변경과 피드 장애로 일부 소스가 실패할 수 있습니다. 다른 소스 결과는 유지되며 status와 doctor로 확인할 수 있습니다.

자동 Story 병합은 제한적입니다

같은 정규 URL은 묶지만 리디렉션 추적이나 제목 유사도만으로 자동 병합하지 않습니다. 오병합을 피하기 위한 선택입니다.

LLM 결과는 ains 결과가 아닙니다

에이전트의 번역·요약·추론에는 오류가 있을 수 있습니다. 중요한 판단은 원문 URL과 ranking.signals로 검증하십시오.

상태 변경 도구를 구분하십시오

수집, 학습 기록, 설정 초기화와 스케줄 설치·해제는 로컬 상태를 바꿉니다. 자동 실행 정책과 사용자 승인을 명확히 하십시오.

v2 전환 승인 전입니다.

실제 24시간 기준점 95%, 7일 기준점 90%, 상위 결과 coverage와 AI 관련 정밀도 등 승인 게이트를 통과하기 전에는 v2를 기본값으로 간주하지 마십시오. AI NEWS HUB와 정확히 같은 순위는 승인 기준이 아닙니다.

14 · TROUBLESHOOTING

문제 해결

증상부터 확인하고 사용자 데이터나 토큰을 공유하지 않은 상태에서 재현 정보를 수집하십시오.

ains 또는 ains-mcp 명령을 찾지 못합니다
  1. npm install -g ai-news-supplier를 다시 실행합니다. 소스 빌드 사용자는 저장소에서 npm install, npm run build, npm link가 성공했는지 확인합니다.
  2. 새 터미널을 열고 PATH를 다시 읽습니다.
  3. Windows는 Get-Command ains, macOS/Linux는 command -v ains로 확인합니다.
  4. 계속 실패하면 MCP 설정에 Node.js와 dist/mcp/server.js 절대 경로를 사용합니다.
MCP가 /mcp 목록에 나타나지 않습니다
  1. .codex/config.toml의 이름과 command = "ains-mcp"를 확인합니다.
  2. 프로젝트가 Codex에서 신뢰된 상태인지 확인합니다.
  3. 앱을 완전히 종료·재실행하고 새 작업을 엽니다.
  4. “ains 소스 상태를 확인해줘”로 get_source_status 실제 호출을 시험합니다.
  5. 표시가 꼭 필요하면 codex mcp add ains -- ains-mcp로 사용자 전역 등록을 검토합니다.

현재 확인된 Codex 앱 사례에서는 프로젝트 범위 서버가 목록에 누락되어도 도구 9개가 실제로 호출되었습니다. 전역 등록은 모든 프로젝트에 영향을 줍니다.

ains-mcp를 실행했는데 아무 출력 없이 멈춥니다

정상일 수 있습니다. STDIO MCP 서버는 사람이 입력하는 셸 명령이 아니라 MCP 클라이언트의 프로토콜 메시지를 기다립니다. 종료하려면 Ctrl+C를 누르고 Codex나 Claude Code 설정을 통해 실행하십시오.

Repo Trending 결과가 없거나 적습니다

24시간·7일 기준점, 별 100개, 최근 14일 push 등의 본선 조건을 확인하십시오. 설치 직후에는 repos/discovery를 사용하고 스케줄 수집으로 실제 snapshot을 축적하십시오.

shell

ains trends --ranking v2 --channel repos --sort discovery
ains doctor

JSON 문법 또는 값 검증 실패 시 경고 후 전체 기본값으로 폴백합니다. ains config show로 현재 유효 설정을 확인하고 따옴표·쉼표와 양수 범위를 점검하십시오.

DB 마이그레이션 또는 무결성 오류가 발생합니다

현재 DB를 덮어쓰거나 삭제하지 마십시오. MCP와 스케줄러를 종료하고 백업 파일을 별도 보존한 뒤 ains doctor 결과와 오류를 확인하십시오. 복구 절차는 “데이터와 보안” 장을 따릅니다.
