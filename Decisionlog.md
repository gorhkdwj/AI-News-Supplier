### D-013 · 승인 게이트 조기 통과 — v2 랭킹 기본 전환을 0.3.0에서 확정 (2026-07-19)

**상황**

- 계약 13절 승인 게이트의 정기 재평가는 7/26 예정이었으나, 7/19 실측에서 잔여 항목이던 24h 기준점 coverage 외 전 항목이 통과됨: 7d coverage 99.1%(기준 90%), community 점수 가용성 100%(기준 90%), precision@20 Repo·Community 각 100%(수동 라벨링, 기준 90%), top20 전원 full coverage, npm pack 통과, 소스 오류 0.
- 24h coverage는 7/18 PC 종료 공백으로 91.1%(기준 95%) → 미러 백필(`fetch --seed`) 후 93.9%. 미달 원인은 측정 시점의 24h 기준 창(now-28h~now-20h)이 공백 구간에 걸린 것으로, 지정 재실측 시각(7/19 23:00) 창으로 계산하면 195/199(98.0%)로 기준을 상회함. 해당 창의 데이터는 미러 병합으로 이미 확정 상태라 23:00 실측도 동일하게 나옴.

**검토한 선택지**

1. 7/26 정기 게이트까지 전환 보류 — 절차적으로 보수적이나, 이미 확정된 데이터로 통과가 계산되는 상태에서 1주 지연.
2. 7/19 조기 통과로 처리하고 0.3.0에서 v2 기본 전환 확정 — publish 전 23:00 공식 재실측 파일로 최종 확인을 조건으로 함.

**결정**

- 2안. 옵션 없는 `ains trends`/`get_trends` 기본을 v2 overview/briefing으로 전환. legacy는 `--ranking legacy` 명시로 한 minor release 유지 후 0.4.0에서 제거(계약 13절 유지).

**근거**

- 게이트 성적표 전 항목 충족. 24h 미달은 데이터 결손이 아니라 측정 창 위치 문제임을 시뮬레이션(out/gate-sim.mjs)으로 확인. 측정 창(7/18 10:00Z~18:00Z)의 스냅샷은 미러 병합으로 확정되어 23:00 결과가 달라질 수 없음.

**영향**

- 옵션 없는 trends 출력이 legacy flat hotness 목록 → v2 4섹션 overview로 변경(Breaking, 0.3.0). legacy는 opt-in으로 강등되고 0.4.0에서 제거 예정. 계약 250행·롤아웃 절, README 양 언어판, docs/index.html, MCP 도구 설명 동기 갱신.

**재검토 조건**

- 23:00 공식 재실측(out/gate-recheck-2026-07-19.txt)이 95% 미달로 나오면 publish 보류 후 재평가. 사용자로부터 v2 precision 저하 신고가 들어오면 랭킹 공식 재검토.

### D-012 · Windows 샌드박스는 unelevated fallback으로 임시 운영 (2026-07-19)

**상황**

- 권장값인 `windows.sandbox = "elevated"`에서 관리자 설정 도우미가 실행되지 않아 로컬 명령이 멈췄음. Codex 0.144.6 업데이트 후에도 동일했으며 unelevated 일회성 테스트는 성공함.

**검토한 선택지**

1. elevated 유지 — 권장 격리는 유지하지만 현재 샌드박스 명령 실행 불가.
2. danger-full-access 사용 — 오류는 우회하지만 프로젝트 파일·네트워크 경계가 사라짐.
3. unelevated fallback 사용 — 전용 저권한 사용자·방화벽 규칙보다 약하지만 restricted token·ACL·오프라인 제어와 기존 read-only 프로젝트 정책 유지.
4. WSL2로 즉시 이전 — 격리 방식은 바뀌지만 현재 Windows 우선 개발 환경과 경로를 크게 변경함.

**결정**

- 3안. 활성 Orca 런타임과 독립 Codex CLI의 `[windows] sandbox`를 `unelevated`로 맞추고, 프로젝트 `.codex/config.toml`의 `sandbox_mode = "read-only"`는 유지함.

**근거**

- 공식 문서가 elevated 설정 실패 시 unelevated를 fallback으로 허용함. 실제 smoke test에서 제한 샌드박스 실행이 복구됐으며 full access보다 최소 권한 원칙에 부합함.

**영향**

- 로컬 명령은 다시 샌드박스 안에서 실행되지만 elevated 모드의 전용 저권한 사용자와 전용 방화벽 규칙보다 격리 강도가 낮음. 프로젝트 코드·제품 동작에는 영향 없음.

**재검토 조건**

- Codex 후속 업데이트 또는 `/setup-default-sandbox`가 elevated 구성을 정상 완료할 때. 복귀 전후에 sandbox smoke test와 `codex doctor`를 모두 통과해야 함.

### D-011 · 릴리스마다 CHANGELOG와 GitHub Release 노트를 남긴다 (2026-07-12)

**상황**

- 사용자가 업데이트 안내를 받아도 "무엇이 바뀌었는지" 알 방법이 커밋 이력뿐이었음. 실사용자(지인) 발생으로 변경 내역 전달 채널이 필요해짐.

**검토한 선택지**

1. 커밋 이력으로 충분 (현상 유지)
2. CHANGELOG.md 파일 + 버전별 GitHub Release 노트 병행

**결정**

- 2안. Keep a Changelog 형식의 CHANGELOG.md(영어, README.md와 동일 기조)를 저장소에 유지하고, publish마다 같은 내용을 GitHub Release(v<버전> 태그)로 게시한다. 0.1.0·0.2.0은 소급 작성.

**근거**

- 업데이트 경험의 3요소(발견-신뢰-실행) 중 "신뢰(변경 내역)"가 공백이었음. CHANGELOG는 저장소·npm에서, Release는 GitHub 알림 구독자에게 각각 닿는 상호보완 채널.

**영향**

- 릴리스 절차에 두 단계 추가: publish 전 CHANGELOG 항목 작성 → publish 후 `git tag v<버전>` + `gh release create`. mirror-data(prerelease)는 버전 릴리스 목록과 분리 유지.

**재검토 조건**

- 릴리스 빈도가 높아져 수동 작성이 부담되면 태그 기반 자동 생성(release-please 류) 검토.

### D-009 · 정적 스냅샷 미러 채택 — 게시 파이프라인 선(先)구현, 클라이언트 시딩 후(後)구현 (2026-07-12)

**상황**

- 로컬 우선 설계상 사용자 기기가 꺼진 동안 스냅샷 공백이 생겨 성장 랭킹(6h 속도, repos trending)이 저하되고, 신규 사용자는 7일 워밍업 전까지 성장 랭킹을 쓸 수 없음. 공식 RSS 약관 조사(W-031)로 미러 범위의 법적 검토는 완료됨.

**검토한 선택지**

1. Supabase 등 완전 서버형 (거부: 비용 스케일링, 운영 책임, Reddit 재배포 리스크, 로컬 우선 헌법 충돌)
2. 현행 유지 + 7/26 게이트 수치 후 미러 결정
3. 미러 채택 — 게시 파이프라인은 지금, 클라이언트 시딩은 v2 전환과 함께

**결정**

- 3안. 근거 데이터는 소급 생성이 불가능하므로 게시를 오늘 시작해 v2 전환 시점(~7/26)에 2주치 미러 데이터를 확보한다. 클라이언트 시딩(`fetch --seed`)은 소비자가 생기는 v2 전환(0.2.0)과 함께 구현한다.

**근거**

- 콜드 스타트 가치는 게이트 수치와 무관하게 항상 유효. 미러는 정적 파일 게시라 서버 비용 0원(공개 저장소 Actions 무료)·무장애 구조 유지. v2 스키마의 결정적 ID와 멱등 upsert 규칙 덕분에 병합이 안전.

**영향**

- 미러 범위: hackernews, devto, github의 Story·Sighting 메타데이터와 metric snapshot만. Reddit(약관: 재배포 제한·48시간 삭제 의무)과 공식 RSS 7종(재배포 명시 허용 없음 + 점수가 없어 미러 실익 0)은 제외 — 제외 소스는 로컬 수집만으로 축적되며 꺼진 동안의 관측은 복구되지 않음(의도된 제한).
- 미러 산출물은 시간별 증분 JSON.gz + sha256 체크섬, 같은 저장소의 rolling release(`mirror-data` 태그)에 게시, 14일 초과 자산은 정리(스냅샷 보존 정책과 일치).
- 기준 계약에 14절(스냅샷 미러) 신설.

**재검토 조건**

- GitHub 무료 한도 정책 변경, 포함 소스의 약관 변경, 미러 데이터 조작 시도 발견 시. 클라이언트 시딩의 quality 표기와 병합 세부는 M3 설계 시 계약에 확정.

### D-008 · v2 기본 전환을 0.2.0 묶음에 합류, publish는 게이트 통과 후로 연기 (2026-07-12)

**상황**

- D-007에서 v2 기본 전환을 0.3.0 이후로 미루고 0.2.0을 개선 묶음으로 확정했으나, 이는 기준 계약 13절("승인 게이트 통과 후 0.2.0에서 v2 기본 전환") 및 v2 계획서 V2-7과 모순됨을 발견. 또한 0.2.0은 아직 publish 전이므로 범위를 다시 정할 수 있음.

**검토한 선택지**

1. 0.2.0을 지금 publish하고 v2 전환은 0.3.0 (D-007 유지, 계약 문서의 버전 표기를 수정)
2. v2 기본 전환을 0.2.0 묶음에 합류시키고, publish를 승인 게이트 통과 후로 연기 (계약 원안 복원)

**결정**

- 2안. 0.2.0 = 기존 개선 묶음(D-007) + v2 Overview 기본 전환. 승인 게이트(계약 13절) 통과 전에는 publish하지 않는다.

**근거**

- 계약 원안이 릴리스 의미를 더 잘 보존함: 0.2.0이라는 minor 승격의 주인공이 핵심 기능(v2)이 됨. 하루 다회 publish 지양(D-007의 묶음 원칙)과도 일치. 사용자가 잦은 버전 업데이트보다 묶음 릴리스를 선호함.

**영향**

- npm 최신 버전은 게이트 통과 시점(~2026-07-26 이후)까지 0.1.0으로 유지됨. Node 20 표기 문제(D-006)도 그때까지 npm에 남으나, 실사용자가 사실상 없어 수용.
- 워밍업 달력 기준일: 상시 수집 시작 2026-07-12 → 워밍업 ~07-19 → shadow 비교 ~07-26 → 게이트 평가.
- legacy 제거는 계약대로 0.3.0.

**재검토 조건**

- 게이트 평가에서 v2가 기준 미달이면 v2 전환만 분리하고 개선 묶음을 먼저 0.2.0으로 publish.
- 그 전에 설치 불능·보안 결함이 발견되면 D-007 예외 조항대로 즉시 patch 릴리스.

### D-006 · 최소 지원 Node 버전을 20에서 22.12로 상향 (2026-07-12)

**상황**

**검토한 선택지**

### D-005 · README와 단일 HTML 사용자 설명서의 역할 분리 (2026-07-10)

**상황**

- 기존 README가 빠른 시작, 전체 명령, 데이터 모델, 마이그레이션, 운영 주의사항을 한 문서에 담아 GitHub 첫 방문자에게는 길고 실제 사용자에게는 탐색성이 부족했습니다.
- 사용자는 README를 빠른 시작·전체 기능 요약·일반적인 GitHub 안내로 유지하고, 상세 사용 설명서는 여러 Markdown 파일이 아닌 보기 좋은 하나의 HTML로 제공하기를 요청했습니다.

**검토한 선택지**

- README 하나에 모든 상세 내용을 계속 유지
- 주제별 Markdown 문서를 여러 파일로 분리
- README는 요약본으로 줄이고 `docs/index.html` 하나를 상세 설명서의 단일 진실 원천으로 운영

**결정**

- 세 번째 안을 채택합니다. README는 5분 빠른 시작, 기능 요약, 설치·기여·라이선스와 상세 설명서 링크만 담습니다.
- 상세 CLI·MCP 사용법, 자연어 작업 예시, 결과 해석, 설정, 데이터 보존, 주의사항과 문제 해결은 자체 CSS·JavaScript를 포함한 `docs/index.html` 한 파일에 둡니다.
- HTML은 외부 CDN이나 빌드 도구 없이 로컬에서도 열리며, 반응형 레이아웃·문서 검색·코드 복사·테마·인쇄 스타일과 접근 가능한 시맨틱 구조를 제공합니다.
- npm 패키지에서도 오프라인 설명서를 열 수 있도록 `docs/index.html`을 패키지 파일 목록에 포함합니다.

**근거**

- GitHub 방문자는 설치와 제품 성격을 빠르게 판단할 수 있고, 실제 사용자는 한 페이지에서 검색과 목차 이동으로 전체 기능을 탐색할 수 있습니다.
- 외부 자산 없는 단일 파일은 npm 배포 전 소스 설치 단계에서도 오프라인으로 열 수 있고 별도 문서 프레임워크의 유지비가 없습니다.
- 상세 설명을 HTML 한 곳에 모으면 README와 여러 문서 사이의 기능 설명 중복을 줄일 수 있습니다.

**영향**

- 기능·옵션이 변경되면 README 요약과 `docs/index.html`의 명령·MCP 표를 함께 검토해야 합니다.
- GitHub 저장소 화면은 HTML을 웹페이지로 직접 렌더링하지 않을 수 있으므로 공개 웹 주소가 필요해지면 GitHub Pages 배포를 별도 승인·검증합니다.

**재검토 조건**

- 단일 HTML의 유지보수가 어려워지거나 다국어·버전별 문서·자동 API 레퍼런스가 필요해지면 정적 문서 생성기 도입을 검토합니다.

### D-004 · Story/Sighting 분리와 유형별 트렌드 랭킹 v2 (2026-07-10)

**상황**

- 단일 `items` 행이 canonical 콘텐츠와 출처별 관측을 동시에 표현하여 같은 공식 발표의 RSS·HN·Reddit 점수와 메타데이터가 덮어써질 수 있었습니다.
- 전역 `hotness = source percentile × 생성 시각 감쇠 × type boost`는 신생 저별 레포를 과대평가하고, 점수가 없는 공식 업데이트와 참여 지표가 다른 커뮤니티를 같은 척도로 비교했습니다.
- AI NEWS HUB 벤치마크에서 레포의 당일 성장, 커뮤니티의 출처별 반응, 공식 업데이트의 최신순 분리가 확인됐지만 비공개 공식·HTML 의존·Reddit 지표 누락은 그대로 복제할 수 없었습니다.

- 기존 전역 공식의 가중치만 조정
- 타입별 공식만 추가하고 단일 item 저장 모델 유지
- 기존 Story ID를 보존하면서 source Sighting·metric snapshot을 추가하고 유형별 랭커로 전환

**결정**

- Repo는 24시간·7일 별 성장과 누적 규모·최근 push, Community는 추천·댓글·증가 속도, Official은 latest 또는 결정적 impact+community echo, Research는 기존 v1 공식을 사용합니다.
- Overview는 점수를 전역 비교하지 않고 네 섹션을 quota로 조합합니다. CLI/MCP 이름과 9개 도구는 유지하고 v2 입력·출력을 추가합니다.

**근거**

**영향**
