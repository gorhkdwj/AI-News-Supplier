### D-006 · 최소 지원 Node 버전을 20에서 22.12로 상향 (2026-07-12)

**상황**

- 0.1.0 공개 직후 구축한 3-OS CI 첫 실행에서 `test (windows-latest, node 20)`만 실패. ① better-sqlite3가 win32+Node20 프리빌트 바이너리를 제공하지 않아 소스 컴파일로 빠졌다 실패, ② 의존성 commander@15가 `engines: node >=22.12.0`을 요구해 우리의 "Node ≥ 20" 선언이 의존성 수준에서 이미 성립하지 않음을 발견.

**검토한 선택지**

1. Node 22.12+로 상향 (engines·문서·CI 매트릭스 갱신, 0.1.1 배포)
2. Node 20 유지 (commander 14.x 다운그레이드 + Windows 빌드 도구 안내 추가)

**결정**

- 1안. `engines.node >=22.12.0`, CI 매트릭스 [22, 24], tsup target node22, doctor 검사 22 기준, README·docs·CLAUDE.md·AGENTS.md 갱신, 0.1.1로 배포.

**근거**

- Node 20은 2026-04 EOL로 보안 패치가 없는 버전이며, EOL 버전 지원을 위해 의존성을 역행(다운그레이드)하는 것은 유지비용 대비 가치가 없음. 프리빌트 공백 문제도 함께 해소됨.

**영향**

- Node 20 사용자는 0.1.1+ 설치 시 engines 경고를 받음(기본은 경고, 차단 아님). 문서·doctor가 22.12 기준으로 일관됨.

**재검토 조건**

- Node 22 EOL 접근 시 동일 절차로 상향. 또는 Node 20 사용자 수요가 실제 이슈로 확인될 때.

### D-005 · README와 단일 HTML 사용자 설명서의 역할 분리 (2026-07-10)

**상황**

- 기존 README가 빠른 시작, 전체 명령, 데이터 모델, 마이그레이션, 운영 주의사항을 한 문서에 담아 GitHub 첫 방문자에게는 길고 실제 사용자에게는 탐색성이 부족했습니다.
- 사용자는 README를 빠른 시작·전체 기능 요약·일반적인 GitHub 안내로 유지하고, 상세 사용 설명서는 여러 Markdown 파일이 아닌 보기 좋은 하나의 HTML로 제공하기를 요청했습니다.

**검토한 선택지**

- README 하나에 모든 상세 내용을 계속 유지
- 주제별 Markdown 문서를 여러 파일로 분리
- README는 요약본으로 줄이고 `docs/index.html` 하나를 상세 설명서의 단일 진실 원천으로 운영

**결정**

- 세 번째 안을 채택합니다. README는 5분 빠른 시작, 기능 요약, 설치·기여·라이선스와 상세 설명서 링크만 담습니다.
- 상세 CLI·MCP 사용법, 자연어 작업 예시, 결과 해석, 설정, 데이터 보존, 주의사항과 문제 해결은 자체 CSS·JavaScript를 포함한 `docs/index.html` 한 파일에 둡니다.
- HTML은 외부 CDN이나 빌드 도구 없이 로컬에서도 열리며, 반응형 레이아웃·문서 검색·코드 복사·테마·인쇄 스타일과 접근 가능한 시맨틱 구조를 제공합니다.
- npm 패키지에서도 오프라인 설명서를 열 수 있도록 `docs/index.html`을 패키지 파일 목록에 포함합니다.

**근거**

- GitHub 방문자는 설치와 제품 성격을 빠르게 판단할 수 있고, 실제 사용자는 한 페이지에서 검색과 목차 이동으로 전체 기능을 탐색할 수 있습니다.
- 외부 자산 없는 단일 파일은 npm 배포 전 소스 설치 단계에서도 오프라인으로 열 수 있고 별도 문서 프레임워크의 유지비가 없습니다.
- 상세 설명을 HTML 한 곳에 모으면 README와 여러 문서 사이의 기능 설명 중복을 줄일 수 있습니다.

**영향**

- 기능·옵션이 변경되면 README 요약과 `docs/index.html`의 명령·MCP 표를 함께 검토해야 합니다.
- GitHub 저장소 화면은 HTML을 웹페이지로 직접 렌더링하지 않을 수 있으므로 공개 웹 주소가 필요해지면 GitHub Pages 배포를 별도 승인·검증합니다.

**재검토 조건**

- 단일 HTML의 유지보수가 어려워지거나 다국어·버전별 문서·자동 API 레퍼런스가 필요해지면 정적 문서 생성기 도입을 검토합니다.

### D-004 · Story/Sighting 분리와 유형별 트렌드 랭킹 v2 (2026-07-10)
**상황**
- 단일 `items` 행이 canonical 콘텐츠와 출처별 관측을 동시에 표현하여 같은 공식 발표의 RSS·HN·Reddit 점수와 메타데이터가 덮어써질 수 있었습니다.
- 전역 `hotness = source percentile × 생성 시각 감쇠 × type boost`는 신생 저별 레포를 과대평가하고, 점수가 없는 공식 업데이트와 참여 지표가 다른 커뮤니티를 같은 척도로 비교했습니다.
- AI NEWS HUB 벤치마크에서 레포의 당일 성장, 커뮤니티의 출처별 반응, 공식 업데이트의 최신순 분리가 확인됐지만 비공개 공식·HTML 의존·Reddit 지표 누락은 그대로 복제할 수 없었습니다.

**검토한 선택지**
- 기존 전역 공식의 가중치만 조정
- 타입별 공식만 추가하고 단일 item 저장 모델 유지
- 기존 Story ID를 보존하면서 source Sighting·metric snapshot을 추가하고 유형별 랭커로 전환

**결정**
- 세 번째 안을 채택합니다. 물리 `items`는 Story aggregate로 유지하고 `source_sightings`, `metric_snapshots`를 additive migration으로 추가합니다.
- Repo는 24시간·7일 별 성장과 누적 규모·최근 push, Community는 추천·댓글·증가 속도, Official은 latest 또는 결정적 impact+community echo, Research는 기존 v1 공식을 사용합니다.
- Overview는 점수를 전역 비교하지 않고 네 섹션을 quota로 조합합니다. CLI/MCP 이름과 9개 도구는 유지하고 v2 입력·출력을 추가합니다.
- 공식 API/RSS만 사용하고 Reddit은 OAuth·식별 User-Agent·삭제 재검증·48시간 보존을 강제합니다. 도구 내부 LLM 호출은 계속 금지합니다.

**근거**
- 출처별 원점수와 토론 URL을 보존해야 성장률·멀티소스 신호·공식성을 정확히 설명할 수 있습니다.
- 타입마다 화제성의 의미가 달라 단일 점수의 가중치 조정으로는 제품 의미 오류를 해결할 수 없습니다.
- ID·FTS·학습 이력은 보존하면서 새 표만 추가하는 방식이 기존 로컬 DB에 가장 안전합니다.

**영향**
- 스키마 v2 백업·마이그레이션과 최소 7일 관측 워밍업이 필요합니다.
- `hotness`는 호환 별칭으로 남지만 다른 `ranking.kind` 간 비교가 금지됩니다.
- 0.1.0은 legacy 기본, 승인 게이트 통과 후 0.2.0에서 v2 기본으로 전환합니다.

**재검토 조건**
- 14일 shadow에서 coverage·precision 게이트를 충족하지 못하거나 Reddit/GitHub 정책·API 계약이 변경될 때 공식과 후보 구성을 재조정합니다.

### D-003 · 학습 기능 방식과 기술 스택 (2026-07-09)
**상황**
- "에이전트가 학습을 설계하게 한다"는 부가 기능의 구현 방식과 전체 스택 결정 필요

**검토한 선택지**
- 학습: (a) 데이터 조회 도구만 제공 / (b) 학습 후보 발굴+지시문 패키징+이력 관리 / (c) 자체 LLM 호출로 학습 자료 생성
- 스택: TypeScript/Node vs Python

**결정**
- 학습은 (b): `get_learning_candidates`, `design_learning_session`, `record_learning`, `get_learning_history` 도구 + MCP 프롬프트. 후보 발굴은 룰 기반(멀티소스 등장, 급상승, 신규성), 도구 자체는 LLM을 호출하지 않음.
- 스택은 TypeScript/Node ≥ 20 (better-sqlite3, @modelcontextprotocol/sdk, commander, tsup, vitest). 패키지명 ai-news-supplier, CLI `ains`.

**근거**
- 지능은 사용자의 에이전트에 있으므로 재료+지시문+이력 관리가 우리 쪽 핵심 가치. 자체 LLM 호출은 키/비용 요구로 로컬 도구 철학과 상충.
- MCP 공식 TS SDK가 가장 성숙하고, npx 한 줄 설치 배포가 가능.

**영향**
- API 키 없이 설치 직후 동작. 학습 품질은 지시문 템플릿과 후보 발굴 룰 품질에 좌우됨.

**재검토 조건**
- 룰 기반 후보 발굴의 정확도가 실사용에서 낮다고 판단될 때(엔티티 사전 확장 또는 옵션형 LLM 분류 검토)

### D-002 · 로컬 우선 아키텍처 + SQLite 축적 + TTL 수집 (2026-07-09)
**상황**
- 수집기와 MCP 서버의 실행 위치(로컬 vs 호스팅)와 수집 시점 결정 필요

**검토한 선택지**
- 실행: 로컬 / 클라우드 호스팅 / 하이브리드 구조
- 수집: 온디맨드만 / 신선도(TTL) 기반 온디맨드 / 백그라운드 상주

**결정**
- 로컬 실행 + 로컬 SQLite(`~/.ai-news-supplier/data.db`) 축적. 코어(src/core)를 통로(MCP/CLI)와 분리해 향후 호스팅 전환 가능하게 설계.
- 수집은 TTL 기반 온디맨드를 기본으로 하고, 선택적으로 OS 스케줄러 등록(`ains schedule install`) 제공.

**근거**
- 실사용 개인/공개 도구 목적에 비용 0원·설치 한 줄·즉시 사용이 최적. 로컬 SQLite 축적으로 호스팅의 장점(장기 데이터) 상당 부분 확보(1년 축적 ~100MB-1GB 수준으로 용량 문제 없음).

**영향**
- 서버 운영 없음. PC가 꺼진 시간대의 실시간성 한계는 소급 수집으로 보완.

**재검토 조건**
- 다수 사용자 공유 또는 끊김 없는 장기 축적이 핵심 요구가 될 때 호스팅 전환 검토

### D-001 · 제품 방향 전환: AI Product Lab → ai-news-supplier (2026-07-09)
**상황**
- 기존 PRD(KDT 해커톤용 PM 특화 AI 학습 서비스)의 가치가 사용자 의도와 불일치

**검토한 선택지**
- 기존 PRD 유지·보완 / PM 요소 제거 후 정보 공급 도구로 전면 전환

**결정**
- PM 관련 요소 전부 제거. 핵심을 "API/크롤링으로 커뮤니티·공식 업데이트·핫레포 수집 → MCP/CLI로 LLM 에이전트에 공급 + 에이전트 주도 학습 설계"로 재정의. 해커톤 제출용이 아닌 실사용 개인/공개 도구로 성격 변경. 기존 docs/PRD/ 문서는 폐기(삭제).

**근거**
- 사용자가 원하는 것은 학습 콘텐츠 플랫폼이 아니라 에이전트에게 최신 AI 맥락을 공급하는 인프라 도구

**영향**
- 웹 프론트엔드/온보딩/개인화 홈 등 기존 P0 범위 전부 무효화. 수집 소스는 최대한 구현(HN, GitHub, 공식 RSS, HF, arXiv, DEV.to + Reddit은 키 입력 시 활성화).

**재검토 조건**
- 없음(전환 완료). 웹 UI가 필요해지면 별도 프로젝트로 검토.
